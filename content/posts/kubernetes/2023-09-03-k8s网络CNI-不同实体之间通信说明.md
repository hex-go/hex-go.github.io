Kubernetes的网络模型是软件定义网络（SDN），原理与硬件物理网络相同，有下面几条规则需要牢记：

- 每个`Pod`都有自己的 IP 地址：不需要在 Pod 之间创建链接，也不需要将容器端口映射到主机端口。
- 不需要`NAT`：节点上的 Pod 无需 NAT 即可与所有节点上的所有 Pod 通信。
- 代理获得全访问通行证：节点上的代理（系统守护进程、Kubelet）可以与该节点上的所有 Pod 通信。
- 共享命名空间：Pod 中的容器共享一个网络命名空间（IP 和 MAC 地址），因此它们可以使用环回地址相互通信。

# K8S网络能解决什么问题

Kubernetes网络是为了确保 Kubernetes 中的不同实体类型能够通信。Kubernetes基础设施之间有很多分隔。命名空间、容器和 Pod 的目的是保持组件之间的区别，因此高度结构化的通信计划非常重要。

<img alt="2023-0903-container-pod-nets.webp" height="90%" src="/images/post-image/2023-0903-container-pod-nets.webp" width="90%"/>

## 容器访问容器网络

容器与容器之间的联网是通过 Pod 网络命名空间实现的。网络命名空间允许你拥有独立的网络接口和路由表，它们与系统的其他部分隔离，独立运行。每个 Pod 都有自己的网络命名空间，Pod 内的容器共享相同的 IP 地址和端口。这些容器之间的所有通信都通过`localhost`进行，因为它们都属于同一个命名空间。(上图中绿线表示）。

## Pod访问Pod网络

在 Kubernetes 中，每个节点都为 Pod 指定了一个 CIDR IP 范围。这确保每个 Pod 都能获得一个唯一的 IP 地址，集群中的其他 Pod 也能看到。创建新 Pod 时，IP 地址永远不会重叠。与容器与容器之间的联网不同，Pod 与 Pod 之间的通信是使用真实 IP 进行的，无论你将 Pod 部署在集群中的同一节点还是不同节点上。

从图中可以看出，要让 Pod 相互通信，流量必须在 Pod 网络命名空间和根网络命名空间之间流动。这是通过虚拟以太网设备或 veth 对（图中 veth0 连接 Pod 命名空间 1，veth1 连接 Pod 命名空间 2）连接 Pod 命名空间和根命名空间来实现的。虚拟网桥连接这些虚拟接口，允许流量通过地址解析协议（ARP）在它们之间流动。

当数据从 Pod1 发送到 Pod2 时，数据流为

1. Pod 1 的流量通过 eth0 流向根网络命名空间的虚拟接口 veth0。
2. 然后，流量通过 veth0 到达虚拟网桥，虚拟网桥连接到 veth1。
3. 流量通过虚拟网桥到达 veth1。
4. 最后，流量通过 veth1 到达 Pod 2 的 eth0 接口。

## Pod访问Service的网络

Pod经常进行动态调度。有时需要根据需求扩大或缩小。在应用程序崩溃或节点故障时，可能会被重新创建。这些事件会导致 Pod 的 IP 地址发生变化。

<img alt="2023-0903-pod-svc-nets.png" height="90%" src="/images/post-image/2023-0903-pod-svc-nets.png" width="90%"/>

Kubernetes 通过使用Service资源解决这一问题，该功能的作用如下：
1. 在前端分配一个静态虚拟 IP 地址，以连接与该服务相关的任何后端 Pod。
2. 将指向该虚拟 IP 的任何流量负载均衡到后端 Pod。
3. 跟踪 Pod 的 IP 地址，这样即使 Pod 的 IP 地址发生变化，客户端也不会在连接 Pod 时遇到任何问题，因为它们只会直接连接到服务本身的静态虚拟 IP 地址。

集群内负载平衡有两种方式：

IPTABLES：在这种模式下，kube-proxy 会观察 API 服务器的变化。对于每个新服务，它都会安装 iptables 规则，捕获服务的集群 IP 和端口流量，然后将流量重定向到服务的后端 Pod。Pod 是随机选择的。由于 Linux Netfilter 无需在用户空间和内核空间之间切换就能处理流量，因此这种模式非常可靠，系统开销也较低。
IPVS：IPVS 建立在 Netfilter 之上，实现了传输层负载平衡。IPVS 使用 Netfilter 的钩子功能，将哈希表作为底层数据结构，并在内核空间工作。这意味着 IPVS 模式下的 kube-proxy 比 iptables 模式下的 kube-proxy 能以更低的延迟、更高的吞吐量和更好的性能重定向流量。


上图显示了从 Pod 1 到 Pod 3 的软件包流经一个服务到另一个节点（红色标记）。前往虚拟网桥的软件包必须使用默认路由（eth0），因为网桥上运行的 ARP 无法理解该服务。之后，软件包必须通过 iptables 过滤，而 iptables 会使用 kube-proxy 在节点中定义的规则。因此，图中显示的是原路径。


## 集群外访问Service

上文已经讨论了如何在集群内路由流量。Kubernetes 网络还有另一方面，将应用程序暴露于外部网络。

<img alt="2023-0903-outside-svc-nets.png" height="90%" src="/images/post-image/2023-0903-outside-svc-nets.png" width="90%"/>

可以通过两种不同的方式将应用程序暴露于外部网络。

Egress：需要将 Kubernetes 服务的流量路由到互联网时，可以使用这种方式。在这种情况下，iptables 会执行源 NAT，因此流量看起来像是来自节点，而不是 Pod。
Ingress：这是从外部世界传入服务的流量。输入还使用连接规则允许和阻止与服务的特定通信。通常，在不同的网络堆栈区域有两种入口解决方案：服务负载平衡器和入口控制器。


## 服务发现

Kubernetes 有两种发现服务的方式：

环境变量：Pod 运行所在节点上运行的 kubelet 服务负责为每个活动服务设置环境变量，格式为 {SVCNAME}_SERVICE_HOST 和 {SVCNAME}_SERVICE_PORT。您必须在客户 Pod 出现之前创建服务。否则，这些客户机 Pod 将无法填充其环境变量。
DNS服务: DNS 服务是作为 Kubernetes 服务实现的，它映射到一个或多个 DNS 服务器 Pod，这些 Pod 的调度方式与其他 Pod 无异。集群中的 Pod 被配置为使用 DNS 服务，其 DNS 搜索列表包括 Pod 自身的命名空间和集群的默认域。集群感知 DNS 服务器（如 CoreDNS）会观察 Kubernetes API 中的新服务，并为每个服务创建一组 DNS 记录。如果整个集群都启用了 DNS，所有 Pod 都能通过其 DNS 名称自动解析服务。Kubernetes DNS 服务器是访问 ExternalName 服务的唯一途径。

## 用于发布服务的ServiceTypes

Kubernetes 服务提供了访问一组Pod的方式，通常是通过使用标签选择器来定义的。这可能是应用程序试图访问集群内的其他应用程序，也可能是让您将集群中运行的应用程序暴露给外部世界。Kubernetes 服务类型（ServiceType）可以让你指定想要哪种服务。

<img alt="2023-0903-svc-type.png" height="90%" src="/images/post-image/2023-0903-svc-type.png" width="90%"/>

不同的服务类型有:

- ClusterIP：这是默认的服务类型。它使服务只能从群集内部到达，并允许群集内的应用程序相互通信。没有外部访问。
- LoadBalancer：此服务类型使用云提供商的负载平衡器对外公开服务。来自外部负载平衡器的流量被定向到后端 Pod。云提供商决定如何进行负载平衡。
- NodePort：通过在所有节点上打开特定端口，允许外部流量访问服务。发送到该端口的任何流量都会转发到服务。
- ExternalName：这种类型的服务通过使用 externalName 字段的内容将服务映射到 DNS，并返回带有其值的 CNAME 记录。不设置任何代理。
